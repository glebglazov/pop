~ Routine
  A recurring unit of unattended agent work — a **Routine schedule** plus a prompt plus a per-Routine **Routine memory** — that fires **Routine run**s over time and never reaches a terminal status. Deliberately not a **Task set**: the Task set lifecycle is terminal (DONE, **Auto-drain clearing**, **Archive**) while a Routine is a generator (ADR-0119). A Routine is **directory-bound**, not repository-bound: creation records the directory it was created from as its bound directory, which may be any directory including non-git ones (e.g. `$HOME`), so **Repository identity** is not required. It lives as a directory artifact `routines/<id>/` in pop's data dir beside `repos/` — manifest (schedule + bound directory), `prompt.md`, `memory/`, `runs/` — which doubles as the **Queue daemon**'s discovery registry. A Routine is **created paused**: authoring alone is not consent to fire — the human proves the flow with manual fires and then resumes it, and the persisted pause bit (`pop routine pause`/`resume`) is the only enable/disable state (no separate draft state). The daemon fires only Routines that are non-paused **and** anchored by at least one prior non-skipped run; a never-fired Routine is never daemon-fired, so the first fire is always a human act. A Routine is editable after creation — prompt and schedule only; the bound directory and id are fixed at creation (change either by delete + re-add). Edits take effect without restarts or gates: the daemon re-reads manifests each tick and the prompt is read at fire time, so a live run keeps the prompt it already read and the next run picks up the new one. A corrupt manifest suspends only that Routine — skipped with a warning on the daemon log and dashboard — never the whole registry. CLI surface: `pop routine add` (scaffolds the dir paused, then drops into the **Routine refinement loop**), `edit` (bare form enters the same loop; `--schedule` stays a direct validated write), `list`, `pause`, `resume`, `fire` (manual immediate run), `runs`, and the **Routine dashboard**. Routines never appear as **Queue dashboard** rows.

~ Routine schedule
  The recurrence declaration in a **Routine**'s manifest, in simple forms only (`every 6h`, `daily at 10:00`, bare-hour `daily at 10` accepted) — not cron syntax. Daily times are **machine-local wall clock** by default; an explicit `utc` suffix (`daily at 10:00 utc`) opts into UTC. Stored instants (fired-at rows) remain UTC; only slot computation converts. Editable after creation via `pop routine edit <id> --schedule "<expr>"` and via the **Routine dashboard**'s edit-schedule action; both paths validate through the same schedule parser before writing, so an unparseable expression is rejected rather than persisted. The **Queue daemon** reads last-fired from the **Execution-state store**; a Routine with no non-skipped run yet is never due (the anchor is the first manual fire). For an anchored routine, a missed fire catches up exactly once immediately (never N times for N missed slots).

+ Routine refinement loop
  The HITL gate a **Routine** is worked in before it earns its schedule: entered automatically by `pop routine add` after scaffolding, by bare `pop routine edit <id>`, and by a refine verb in the **Routine dashboard** action menu. Single-key verbs between iterations: `c` opens a **Routine authoring session**, `f` fires a real test run streaming in the terminal (a normal **Routine run** — it records a run row, keeps its report, and becomes the schedule anchor; there is no dry-run variant), `v` opens the last report in `$PAGER`, `e` opens `prompt.md` in `$EDITOR`, `s` edits the schedule, `r` resumes the Routine and exits, `q` exits leaving it paused. The loop is how the created-paused discipline resolves: iterate until a manual fire looks right, then resume.
  avoid: wizard, setup mode

+ Routine authoring session
  The interactive agent chat spawned from the **Routine refinement loop**'s `c` verb: pop launches the agent in the Routine's bound directory (so it can probe the repo and any MCP tooling, e.g. live JQL) with front-loaded rules embedded in the pop binary — the routine framework contract (prompt wrapping, **Routine memory** dir, report path, schedule grammar including local/utc), the Routine's concrete paths, and an interview checklist (goal, data source, definition of seen/new, memory format, report format, empty-run behavior). The agent edits `prompt.md` directly and changes the schedule only through `pop routine edit <id> --schedule` so parser validation holds. Preset resolution follows `[routines].agents`; a `--agent <preset>` flag on `add`/`edit` overrides it for that loop session's chats.
  avoid: prompt wizard

~ Routine dashboard
  The interactive `pop routine dashboard` TUI — sibling view of the **Queue dashboard** in the same shell, toggled bidirectionally with `v`; each command opens its own view. One row per **Routine**: ROUTINE, DIRECTORY, SCHEDULE, LAST RUN, STATUS (paused/running/idle). Keys mirror the queue grammar, including its action-menu pattern: `a` opens a layered action overlay on the focused row whose verbs are fire, pause/resume, preview, edit prompt (suspends the TUI into `$EDITOR` on `prompt.md`), edit schedule (a modal text input pre-filled with the current schedule, validated on enter with inline error and re-edit), refine (suspends the TUI into the **Routine refinement loop**), and runs. Direct shortcuts stay for `i` fire now, `p` preview the run pane, `l`/Enter open the runs-list detail view (open a report from there); pause-toggle lives only in the action menu (its former direct `a` binding is the menu opener now). `h`/left/esc back, `gg`/`G` top/bottom. A Routine whose manifest fails to load renders as a warning row rather than erroring the whole view.
