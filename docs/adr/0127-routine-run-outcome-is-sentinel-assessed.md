# Routine run outcome is sentinel-assessed, not exit-status-inferred

A Routine run's outcome used to be inferred from the agent's exit status alone: exit 0 → succeeded, anything else → failed. Headless agents routinely exit 0 having silently failed the job — no report written, work half-done — so "succeeded" rows lied, and the dashboard could never honestly answer "which routines passed". We considered report-existence as the success signal and rejected it as too weak (an agent can write an empty or apologetic report and still exit 0); exit-status-only was rejected for the same reason from the other side.

Instead routines adopt the tasks completion-sentinel contract (`tasks/assess.go`): the run postamble that pop already wraps around `prompt.md` now requires the agent to end its output with `ROUTINE_COMPLETE` or `ROUTINE_FAILED: <reason>`. The names are routine-specific, not reused `TASK_*`, so a routine prompt that happens to discuss tasks can't false-positive. Assessment ladder: exit 0 + `ROUTINE_COMPLETE` + report on disk → succeeded; `ROUTINE_FAILED` → failed with the sentinel's reason as `fail_reason`; exit 0 with no sentinel → failed (missing sentinel); no report file → failed (missing report); nonzero exit, exec error, crash-reconcile, and quota-exhaustion remain fail reasons as today.

Consequences: an agent that completes cleanly but skips the sentinel is recorded failed — deliberate, matching the tasks discipline where the sentinel is the completion contract, not decoration. Fail reasons become first-class display material: the runs detail view shows them, and the dashboard STATUS column can honestly show ok/failed for idle routines. Under [ADR-0128](0128-any-failure-or-run-affecting-change-pauses-a-routine.md), a sentinel-assessed failure also pauses the routine.
