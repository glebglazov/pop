# Wayfinder Maps are a first-class sibling of Task sets

Pop adopts the wayfinder skill (forked with changes from mattpocock/skills, like the other embedded planning skills per [ADR-0009](0009-planning-skills-are-embedded-in-the-binary.md)) to plan efforts too big for one agent session: a **Map** of **Decision tickets** worked one at a time until the way to a destination is clear. The obvious home was the Task-set machinery — a map is a set of markdown work items with `Blocked by` edges — and we rejected it. A Map violates the Task-set contract on every load-bearing clause: its membership grows and shrinks continuously as fog graduates into tickets (sets are fixed at registration), its tickets need a persisted `claimed` state (persisted `in_progress` is a contract fault for tasks), its resolutions are prose answers written into Task storage rather than agent commits against a runtime checkout (the implement done-condition), and a Map is HITL-dominant by design, so as a Task set it would sit BLOCKED on every dashboard forever. Teaching implement/queue/status a second "kind" of set would have spread wayfinding semantics through the manifest contract, the status derivation, and the drain path.

Instead a Map is its own concept beside Task sets: stored per-repository in Task storage under a `wayfinder/` sibling of `tasks/` (`<timestamp>-<slug>/map.md` + `issues/NN-<slug>.md`, plain markdown, `Type:`/`Status:`/`Blocked by:` lines), existing because its folder exists — no registration, and the Queue daemon never sees it. Its lifecycle is declared, not derived: a `Status:` line in `map.md` (`active`/`done`/`abandoned`), because "the way is clear" is a judgment the session records, plus a pop-side reversible Archive flag for cleanup, mirroring Task-set Archive. The two concepts connect by a forward link only: when the way (or an early-splittable chunk) is clear, to-prd/to-tasks spawn ordinary registered Task sets; the Map records the spawned set ids and the set's `prd.md` records its source Map. One Map may spawn many sets.

We also considered reusing Matt's skills verbatim behind his `/setup-matt-pocock-skills` tracker-doc seam (pop as a configured "tracker"). Rejected for now: that seam is per-repo by construction, and pop's default is zero repo files with global configuration; closing that gap cleanly for all five agents wasn't worth blocking wayfinder adoption. The fork keeps upstream's vocabulary (map, destination, fog, frontier, decision ticket) so skill prose and CLI speak one language — but note the deliberate rename pressure-point: upstream calls the whole endeavour an "effort", which pop cannot, because **Effort** is already the task strength knob.
