# Routines are created paused and anchored by a manual first fire

Routines used to be born unpaused, and a never-fired Routine counted as due immediately — so `pop routine add` followed by a daemon tick would fire the scaffolded stub prompt as a junk agent run before the author had written anything. We considered softer fixes (keep immediate firing but rely on the author editing fast; or compute the next future slot for a never-fired routine so it at least waits until its scheduled time) and rejected both: neither guarantees a human has ever seen the routine actually work before it runs unattended.

Instead the discipline is human-proof-first: `pop routine add` writes `paused: true`, and the Queue daemon fires only Routines that are non-paused **and** anchored by at least one prior non-skipped run row. A never-fired Routine is never daemon-fired under any schedule — the first fire is always a human act (`pop routine fire`, or `f` in the [Routine refinement loop](0125-routine-refinement-is-an-in-cli-hitl-loop-with-a-front-loaded-authoring-agent.md)), and that run row becomes the schedule anchor. This is a convention, not a hard gate: `resume` is never refused, but resuming an unanchored routine still fires nothing until a manual run anchors it.

Consequences: the natural authoring flow (create → iterate → manual fire → resume) lands the next daemon fire on the next scheduled slot after the manual run. Catch-up-once on stale resumes is deliberately kept — an anchored routine resumed after a long pause fires once immediately, which the manual-fire-then-resume flow never hits.
