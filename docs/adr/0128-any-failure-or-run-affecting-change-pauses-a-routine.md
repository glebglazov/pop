# Any failure or run-affecting change pauses a routine

[ADR-0124](0124-routines-are-created-paused-and-anchored-by-a-manual-first-fire.md) made routines human-proof-first at birth: created paused, anchored by a manual fire. But the discipline stopped at creation — an edited prompt fired on the next slot untested, and a routine that started failing kept firing on schedule. Both violate the same principle: a human must have seen the current shape of the routine work before it runs unattended, because the discouragement of a routine failing on schedule is bigger than the friction of re-proving it. We considered keep-firing-and-show-red (rejected: a broken routine burning agent runs unattended is the exact failure mode ADR-0124 exists to prevent), pause-after-N-consecutive-failures (rejected: N−1 junk runs buy nothing), and enforcing edits only at CLI write chokepoints (rejected: the refinement-session agent and stray `$EDITOR` sessions edit `prompt.md` directly, bypassing every chokepoint).

The decision is two triggers, one state. **Failure pauses:** any Routine run that finishes failed — daemon-fired or manual — sets `paused` with reason `failure`. **Change pauses:** editing any run-affecting input (prompt, schedule, runtime agent list, effort) sets `paused` with reason `changed`. Change detection is belt-and-braces: write chokepoints (CLI flags, dashboard modals, edit-prompt verb) pause eagerly for immediate feedback, and a **run-affecting fingerprint** — a canonical hash of the explicitly-set run-affecting inputs, recorded on every run row — is the safety net: the daemon compares the current fingerprint against the last run's before firing and on mismatch pauses instead of fires, which catches direct `prompt.md` edits no chokepoint saw. Unset fields are omitted from the canonical form, so adding a new run-affecting criterion in a future version never mass-pauses existing routines — a fingerprint only moves when a human sets something. The pause reason (`created`/`manual`/`failure`/`changed`) is persisted and surfaced by the dashboard and refinement loop.

Consequences: the authoring flow is unchanged (create → iterate → manual fire → resume), and the same loop now also covers "I tweaked the prompt" and "it broke overnight" — fix, fire manually (which records the new fingerprint and, per ADR-0124, re-anchors nothing because the anchor is kept), resume. As with ADR-0124 this is a convention, not a hard gate: `resume` is never refused, and resuming without a proving fire is on the human.
