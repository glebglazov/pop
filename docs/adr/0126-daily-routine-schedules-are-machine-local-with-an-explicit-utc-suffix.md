# Daily routine schedules are machine-local with an explicit utc suffix

The original implementation was accidentally timezone-inconsistent: `daily at HH:MM` slots were computed in the location of the stored last-fired instant (UTC, since fired-at rows are stored RFC3339 UTC) but in `time.Local` on the never-fired branch — so "daily at 11" effectively meant 11:00 UTC without anyone having decided that. We decided daily times are **machine-local wall clock** by default, because that is what a human writing "daily at 11" means; an explicit `utc` suffix (`daily at 11:00 utc`) opts into UTC, and the bare-hour form (`daily at 11`) is accepted. The alternative — UTC default with a display hint — was rejected as the wrong default for a single-user, single-machine tool.

Stored instants stay UTC everywhere (store rows, manifests' created-at); only slot computation converts into the schedule's zone. This silently flips the meaning of pre-existing `daily at` manifests from UTC to local — accepted deliberately: at decision time one routine existed and had never fired, so no migration was written. DST is inherited from the local zone: a slot is whatever the wall clock says that day.
