# Routine refinement is an in-CLI HITL loop with a front-loaded authoring agent

Writing a Routine prompt blind is hard: the author has to intuit the wrapping contract (memory preamble, report postamble), invent a memory format, and guess whether the data source even works — and the old flow (`add` opens `$EDITOR` on a three-line stub) gave no way to iterate. We considered a richer static prompt template and an external authoring skill, and rejected both in favour of building the iteration loop into pop: templates can't answer questions or probe the data source, and a skill wouldn't ship with the binary or bind to the concrete routine paths.

`pop routine add` (after scaffolding, replacing the bare editor open) and bare `pop routine edit <id>` enter a single-key HITL gate — `c` chat, `f` fire test run, `v` view last report, `e` edit prompt in `$EDITOR`, `s` edit schedule, `r` resume + exit, `q` quit paused — and the Routine dashboard's action menu gains a refine verb into the same loop; no new sibling subcommand (first-letter tab-completion is kept clean). `c` spawns an interactive agent in the Routine's bound directory — where repo context and MCP tooling live, so it can test a JQL query for real — front-loaded with rules embedded in the pop binary: the framework contract (prompt wrapping, memory dir, report path, schedule grammar including local/utc), the routine's concrete paths, and an interview checklist (goal, data source, definition of seen/new, memory format, report format, empty-run behavior). The agent edits `prompt.md` directly but changes the schedule only through `pop routine edit --schedule`, so parser validation is never bypassed. Preset resolution follows `[routines].agents`; `--agent <preset>` on `add`/`edit` overrides it for that session's chats.

Fires from the loop are real runs — recorded, report kept, anchor set (see [ADR-0124](0124-routines-are-created-paused-and-anchored-by-a-manual-first-fire.md)); a dry-run variant was considered and rejected — iterate by re-firing.
