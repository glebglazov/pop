# The Routine refinement loop is tmux-spawned whole from the dashboard, and its session is mode-aware

[ADR-0125](0125-routine-refinement-is-an-in-cli-hitl-loop-with-a-front-loaded-authoring-agent.md) had the Routine dashboard suspend into the refinement gate via `tea.Exec` and the gate's agent-session choice spawn an interactive agent sharing pop's terminal. Reached from the dashboard that agent renders but never receives a keystroke.

The cause is process-group handling in pop, not bubbletea. Two dashboard verbs are the control experiment: the Work dashboard's shell verb and the Routine dashboard's edit-prompt verb suspend the same program onto the same tty via `tea.ExecProcess` with a bare `exec.Cmd`, and both take keyboard input correctly. They pass no `SysProcAttr`, so the child inherits pop's process group — already the terminal's foreground group — and nothing moves. The refinement session is the only interactive child in pop that goes through `tasks.RunAttended`, which sets `SysProcAttr{Foreground: true, Ctty: 0}`; per `syscall/exec_libc2.go:117-141` the child `setpgid`s out of pop's group *first* and only then calls `tcsetpgrp`, i.e. from a background process (Go relies on the post-fork signal mask to keep the resulting `SIGTTOU` pending), and it aims that `tcsetpgrp` at `Ctty: 0` — pop's fd 0 — while the surrounding code reads the old foreground group from `ttyFd`, bubbletea's separately-opened `/dev/tty`. Two descriptors, assumed interchangeable. Which link in that chain yields "renders but no keys" is not pinned down, but the class is: only this path changes process groups, and only this path fails.

The bubbletea input reader is *not* the cause, despite surviving `releaseTerminal` (which cancels it but, unlike `shutdown`, never `Close()`s it). It survives parked on a channel send into `p.msgs`, because `p.exec` blocks the event loop — and a goroutine parked on a send issues no reads, so it takes keystrokes from nobody. That is consistent with the two working verbs, and it means upgrading bubbletea would fix nothing (`exec.go` and `tty.go` are byte-identical from v2.0.0 through v2.0.6, `cancelreader` pinned at v0.2.2 in both) and forking it over one missing `Close()` would fix nothing either.

So the dashboard stops sharing its terminal at all: **the refine verb spawns the entire refinement loop into its own tmux window** — `pop routine edit <id>` (with `--refine-agent` forwarded when set) run in a window named after the Routine id, in the tmux session derived from the Routine's bound directory, reused and switched to when one already exists, mirroring the wayfinder spawn. The gate and the agent it launches both run in that window's own pty, where pop is a plain CLI process with a single tty owner — the configuration that already works. `tea.Exec` and the `refineExec` adapter disappear from the Routine dashboard. We considered the narrower repairs the diagnosis admits — matching the two working verbs by dropping `RunAttended` for a plain `Run` on this path, or pointing `Ctty` at `ttyFd` instead of fd 0 — and chose not to rest the feature on them: they keep an interactive agent inside a suspended TUI, which pop has already declined to do everywhere else (Drains, Routine runs, and wayfinder maps all spawn into tmux). We also considered spawning only the *agent* into tmux while the gate stayed under `tea.Exec` (leaves the gate itself sharing the terminal for no gain), and quitting and relaunching the TUI around an inline session (sound, but strictly more machinery than spawning a window). Outside tmux the dashboard path refuses and names `pop routine edit <id>` rather than falling back to the inline suspend, which would preserve the broken path as a silent trapdoor.

Consequences: the dashboard no longer blocks on refinement — it stays live while the loop runs elsewhere, so its rows must refresh from the store rather than from a gate return. Because the whole loop moves, the gate's agent-session choice stays *blocking within the pane* exactly as on the CLI, so firing a test run mid-edit is impossible by construction rather than by policy. A refine on a Routine whose window already exists switches to it and sends nothing, never typing a command into a live gate or agent. The `Ctty: 0`/`ttyFd` mismatch in `tasks.RunAttended` is fixed regardless, on its own merits — it still governs every CLI-spawned attended agent (drain gates, the CLI refinement loop), where a caller whose fd 0 is not the tty makes `cmd.Start()` fail outright.

Independently, the refinement session is now **mode-aware**. Its front-loaded rules were unconditionally greenfield ("interview me and write a good prompt.md"), so refining an existing Routine re-ran the whole new-routine interview. The session now branches on the content of `prompt.md` — the `add` stub or blank means create, anything else means revise — rather than on the call site (an abandoned first `add` session would then lie) or on run history. In revise mode the current `prompt.md` is embedded **verbatim** in the front-loaded prompt and the six-item checklist is reframed as an audit: identify what the prompt already settles, ask only about what the human wants changed or what the prompt leaves ambiguous. Embedding beats pointing at the path because obedience to "go read this file" is a per-agent gamble the briefing should not take; nothing else (last report, memory contents, run history) is embedded — the session runs in the bound directory and can read them itself.
