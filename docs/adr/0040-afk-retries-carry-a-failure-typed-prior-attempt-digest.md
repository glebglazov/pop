# AFK retries carry forward a stream-derived, failure-typed digest of prior attempts

> **Relates:** qualifies ADR 0020's "the structured reason is the durable signal" for the unattended retry loop; extends the **Captured attempt stream** of ADR 0016 from a read-only telemetry substrate to a prompt input

A **Task attempt** that fails is retried with the runtime checkout's files preserved but with a byte-identical prompt: `BuildAgentPrompt` carries no memory of the prior attempt, so attempt N+1 re-discovers everything from scratch and is free to repeat attempt N's mistake. The attended gates already feed history (the HITL assistance prompt summarizes completed AFK work; the Failed gate reads the last attempt's footer reason), but the **unattended worker** — the one actually looping — flies blind. The concrete failure that motivated this: a long-running command (a Playwright suite) backgrounded and polled in a single non-interactive `claude -p` session, where the agent yields its turn to "wait", the session ends with no **Completion sentinel**, the attempt is assessed failed, and a fresh retry repeats it — `FAILED → RESET → FAILED`.

Decision: on a retry, the worker prompt carries forward two distinct feeds, each drawn from the store designed for it (the **State / Journal / Telemetry** split). **Own prior-attempt story (intra-task)** comes from this task's **Captured attempt stream** files: the footer `(outcome, reason)` is mapped to a *failure type* — contract failure (missing sentinel/summary, unchecked acceptance, timeout) → **"continue"**; the agent's own `TASK_FAILED:` → **"pivot/reassess"**; a crash exit → **"reassess"** — and the assistant text + tool ticks supply the approach narrative, scoped to attempts **since the last Open-task reset**. **Sibling completed briefs (inter-task)** come from the **Progress record** via the existing completed-AFK join (manifest status `done` + `DONE`/`COMPLETE` outcome), now deduped to the **latest record per task**, and are fed to the worker as they already are to the HITL assistance prompt. A companion clarification to `BuildAgentPrompt` states the attempt is a single non-interactive session with no later turn — ending without a sentinel is a failure, and long work must be polled across successive bash calls rather than waited on by yielding — which is the actual fix for the yield; the carry-forward makes the retries that remain converge instead of repeat.

## Considered options

- **Feed the whole Journal (`progress.txt`) to the worker.** Rejected. The Journal is terminal-grain: a failed task is one line (`failed after N attempts: <reason>`) with no record of the *approach* that failed. The "try a different angle" signal the carry-forward exists to provide is physically absent from the Journal — it lives only in the stream — so a whole-journal feed could not deliver it. It also grows with set size and would feed sibling failure/reset churn that teaches a worker nothing about unrelated files.
- **A flat "this attempt failed", untagged.** Rejected. The dominant real failure is a *contract* failure on a *sound* approach (the suite was running; the agent just yielded). An untagged "it failed" teaches the next attempt to abandon the correct path; the failure-type tag is what routes continue-vs-pivot.
- **Require the agent to write its own status block for the next attempt.** Rejected. The same agent that drops the Completion sentinel will drop a status obligation. The digest is derived harness-side from the stream the harness already owns — no new agent contract.
- **Harness resume of the same session on a missing sentinel.** Deferred, not rejected. Continuing the same `claude` session (via the session id already present in the stream's `init` event) would remove the yield failure regardless of model compliance — strictly more robust than a prompt clause. But it redefines a Task attempt from "one session, retried fresh" into "a resumable conversation", needs session-id lifecycle plumbing, and changes what a stream file represents. Revisit when there is appetite; it would carry its own ADR.

## Consequences

- `BuildAgentPrompt` becomes non-deterministic across attempts — it now depends on prior attempts' streams and siblings' journal summaries. This is the deliberate trade for convergence and is fully reversible (stop injecting); nothing about it is persisted.
- ADR 0020's invariant holds unchanged: the worker never receives a pointer to a raw **Captured attempt stream**, only a harness-built digest. The stream stays the authoritative substrate (ADR 0016); the retry digest is one more derived lens, like the **Attempt timing breakdown**.
- `completedAFKProgress` gains a dedupe-to-latest-record-per-task step and a second caller (the worker prompt, alongside the HITL assistance prompt).
- Carry-forward fires only on a real retry (a fresh session), never within an attempt. The cut at the last Open-task reset mirrors the sibling dedupe-to-latest: both let State / the reset mark what is stale, since a human reopens precisely because the prior line of attack was abandoned.
